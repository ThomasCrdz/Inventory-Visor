1. Arquitectura General

El aplicativo es un SPA (Single Page Application) monolítico contenido en un único archivo HTML. Toda la lógica, estilos y markup conviven en el mismo fichero sin dependencias locales externas — solo una librería vía CDN:

DependenciaVersiónFuenteUso
xlsx.full.min.js0.18.5cdnjsLeer/escribir archivos Excel (XLSX)
JetBrains Mono—Google FontsFuente monoespaciada (datos, labels)
Syne—Google FontsFuente UI (títulos, tabs, botones)

Modelo de capas

┌─────────────────────────────────────────────────┐
│  UI / HTML (estructura visual)                  │
│  ├── NAV (tabs)                                 │
│  ├── PAGE: Exportar                             │
│  └── PAGE: Visor                                │
├─────────────────────────────────────────────────┤
│  CSS (design system custom con CSS variables)   │
├─────────────────────────────────────────────────┤
│  JavaScript (lógica de negocio inline)          │
│  ├── Tab Navigation                             │
│  ├── Export Module (CSV parser + procesador)    │
│  └── Visor Module (filtros + tabla + paginado)  │
└─────────────────────────────────────────────────┘

Páginas (pestañas)

La navegación entre páginas no usa routing — simplemente alterna display: block/none mediante clases CSS .page.active. Hay dos páginas:

  • Exportar — carga CSVs, cruza datos, genera XLSX
  • Visor — carga XLSX, filtra y visualiza datos con tabla paginada

4.2 parseCSV(text) → Array<Object>

Propósito: Convertir texto CSV crudo en un array de objetos JavaScript, con tolerancia a variantes de formato de distintas exportaciones.

Flujo de parseo:

  1. Strip BOM — elimina ﻿ (Byte Order Mark de archivos generados en Windows)
  2. Detección de separador:
    • Si la primera línea comienza con sep= → usa el carácter indicado y descarta esa línea
    • Sino: cuenta ocurrencias de ; vs , en la primera línea — el más frecuente gana. Default: ,
  3. parseRow(line) — parser de fila con soporte de campos quoted:
    • Mantiene estado inQ (inside quotes)
    • Maneja comillas dobles escapadas "" → "
    • Acumula cur y lo empuja al array al encontrar el separador fuera de comillas
  4. Sanitización cv(v) — elimina prefijos = (artefacto de celdas de texto en Excel), comillas externas, y trim
  5. Headers extraídos de la línea start (0 o 1 si había directiva sep=)
  6. Filas: salta líneas vacías o con todos los valores en blanco

4.3 loadSource(key, inp)

Propósito: Manejar la selección y carga de un archivo CSV para una fuente específica.

  • Extrae el File del HTMLInputElement
  • Crea FileReader, lee como texto UTF-8
  • En onload: llama parseCSV(), almacena en raw[key]
  • Actualiza UI: clase del card (.fc.ok o .fc.err) y texto de estado con conteo de filas
  • Agrega entrada al log

4.4 normKey(v) → string

const normKey = v => (v || '').replace(/\D/g, '');

Propósito: Normalizar IMEIs/ICCs/seriales eliminando todo carácter no numérico. Permite comparar valores que tengan guiones, espacios o distintos formatos de presentación entre fuentes.

Ejemplo: "867-929 064346770" → "867929064346770"

4.5 buildImeiMap(rows, col) → Map<string, Object>

Propósito: Construir índice de lookup por IMEI/serial para búsqueda en tiempo O(1) en lugar de O(n) por cada fila.

  • Para cada fila normaliza el valor de col con normKey
  • Solo indexa valores con 10+ dígitos (descarta vacíos o truncados)
  • Retorna Map<normalizedKey, rowObject>

4.6 buildSimMap(simsRows, podRows) → Map<string, string>

Propósito: Construir índice combinado IMEI normalizado → ICC/SIM fusionando dos fuentes con prioridad definida.

FuentePrioridadColumnas CSV
SIMs AT&T (sims)Alta — se inserta siempreIMEI → ICC
POD (pod)Baja — solo si IMEI no está en AT&TIMEI → Subscriber ID # o MSISDN

4.7 startProcess()

Propósito: Orquestar el procesamiento completo de manera no bloqueante usando chunking con setTimeout.

Validación previa: Requiere al menos raw.unidades o raw.inventory. Si ninguno cargado → toast de error.

Flujo asíncrono:

  1. Deshabilita botón, muestra progress bar, agrega log de inicio
  2. setTimeout(..., 50) → cede control al navegador para renderizar los cambios UI
  3. Construye los 5 índices de lookup: invMap, invImei, invVeh, hwMap, simMap
  4. Define función recursiva chunk():
    • Procesa 500 registros por tick
    • Actualiza progress: (i/total × 85) + 10 (reserva 10% inicio, 5% finalización)
    • Si quedan registros: setTimeout(chunk, 0) (cede al browser entre chunks)
    • Si terminó: llama finalize()
Por qué chunking
El procesamiento de archivos grandes (>10k filas) bloquearía el thread principal, congelando la UI. El chunking con setTimeout cede el control al event loop entre cada bloque de 500 registros, manteniendo la interfaz responsiva y actualizando la barra de progreso en tiempo real.

4.9 finalize()

Ejecutado después de procesar todos los registros:

  1. Mueve progress bar a 100%, mensaje "¡Listo!"
  2. Calcula estadísticas: total, con serial, con SIM Guardian, con IMEI GPS
  3. Actualiza DOM: stats box, botones Exportar/Ver Visor/Limpiar, badge del nav
  4. Agrega log de éxito con conteo total
  5. Dispara toast de confirmación
  6. Con setTimeout(..., 700ms) → resetea la progress bar

5.5 buildSidebar()

Genera dinámicamente el HTML del sidebar a partir de FILTER_DEFS. Llamado una vez al cargar la página (no por cada carga de datos). Cada grupo incluye: label con icono, <textarea> con id fta-{key} y handler oninput, y hint de uso.

5.6 parseTerms(val) → Set<string>

Convierte el valor de un textarea de filtro en un Set de términos normalizados:

  • Split por \n o , (soporta multi-línea y separados por coma)
  • Trim + lowercase
  • Filtra strings vacíos

Ejemplo: "PSVR54\nPSVR56, RLLW48" → Set{"psvr54", "psvr56", "rllw48"}

5.7 applyVFilters()

Lógica de filtrado multi-campo:

Comportamiento lógico
AND entre campos: Cada campo con términos activos DEBE tener al menos un match.

OR dentro de un campo: Si "vehicle" tiene términos ["psvr54", "psvr56"], la fila pasa si vehicle contiene "psvr54" OR "psvr56".

Match type: substring case-insensitive con String.includes()

Sort: Si vSortCol !== null, ordena vFiltered por VCOLS[vSortCol].key con comparación string lowercase y dirección vSortDir.

Llama renderVTable() y updateVInfo() al finalizar.

5.8 renderVTable()

Renderiza la tabla paginada completa inyectando HTML en #tablePanel.

  1. Si vFiltered vacío → renderiza empty state con mensaje contextual
  2. Calcula slice: start = (vPage-1) × vPageSize, end = min(start+vPageSize, total)
  3. Construye mapa termsByCol para highlight
  4. hlCell(val, colKey):
    • Celda vacía → <span class="empty">—</span>
    • Columna genGuardian → span con clase .gen1/.gen2/.gen3 (amber/cyan/green)
    • Busca término activo con indexOf(t) case-insensitive
    • Si match → envuelve texto coincidente en <mark class="hl">
  5. Genera tabla completa + pager

5.9 buildVPager(total) → string

Genera el HTML del paginador. Para ≤7 páginas muestra todas; para más, usa algoritmo de ellipsis mostrando [1, …, vPage-2..vPage+2, …, last]. Botones «‹›» de navegación con estados disabled en extremos.

5.10 Carga de archivos en Visor

Drag & Drop: Los handlers vDragOver, vDragLeave, vDrop gestionan el arrastre visual y extraen el archivo de e.dataTransfer.files[0].

vReadFile(file): Usa FileReader.readAsArrayBuffer() (necesario para XLSX). En onload: XLSX.read(data, {type:'array', raw:false}) → toma primer sheet → sheet_to_json({header:1}) (array 2D sin interpretar) → pasa a vParseRows().

vParseRows(raw, fname):

  1. Detecta fila de headers: itera primeras 5 filas buscando la que tenga ≥3 columnas reconocidas por COL_ALIASES
  2. Mapea cada columna a su key interna vía COL_ALIASES
  3. Construye vAllRows con objetos de las 13 keys de VCOLS
  4. Salta filas completamente vacías (every(v => v === '' || null))
  5. Llama clearFilters() → oculta drop zone → muestra layout → actualiza badge → applyVFilters()

5.11 exportVFiltered()

Exporta a XLSX solo los registros del filtro activo (no toda la data). Mismo formato que doExport() pero sheet name 'Filtrado'. Nombre del archivo incluye sufijo _filtrado_{N} o _completo según si hay filtros activos.

6.1 Flujo Export

SECCIÓN 8

8. Consideraciones Técnicas y Limitaciones

8.1 Procesamiento 100% client-side

Ventajas: Sin servidor, sin backend, sin red — funciona offline. Los archivos CSV/XLSX nunca salen del dispositivo del usuario.

Limitaciones:

  • El navegador puede quedarse sin memoria con archivos muy grandes (>100k filas)
  • Sin persistencia: al refrescar la página se pierde todo el estado
  • El procesamiento está limitado a un solo thread (Web Worker no implementado)

8.2 Tolerancia a formatos de CSV

  • BOM (﻿) en archivos Windows
  • Separadores , o ; (auto-detección por frecuencia)
  • Directiva sep= de Excel
  • Prefijos = de celdas numéricas exportadas desde Excel
  • Campos con comillas conteniendo comas internas
  • Comillas dobles escapadas ("" → ")

8.3 Normalización de IMEIs

Todos los lookups por IMEI/ICC/serial numérico pasan por normKey() que elimina caracteres no numéricos. El criterio de validez mínima es 10 dígitos (descarta valores nulos o truncados por errores de formato).

8.4 Prioridad de fuentes

Para cualquier campo dado, el orden de prioridad de fuentes es:

  1. CSV de Unidades sin comunicar (fuente principal)
  2. Inventory Report (enriquecimiento por serial > IMEI > vehículo)

Para SIM cards: AT&T → POD. Para SIM GPS: Hardware → simMap general.

8.5 Seguridad

  • esc() previene XSS al renderizar contenido de archivos externos en la tabla

Aclaración sobre el stack
No hay "Java" en la ecuación. El stack es React + TypeScript + CSS (Tailwind). TypeScript es un superset de JavaScript que agrega tipado estático — compila a JavaScript normal.

shadcn/ui no es un MCP — es una librería de componentes React que se instala en el proyecto con npx shadcn@latest init. No requiere herramienta externa.
